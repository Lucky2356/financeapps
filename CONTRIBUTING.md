# Вклад в проект

Спасибо за интерес к **Финансовому помощнику**! Любой вклад — код, баг-репорты,
документация, идеи — приветствуется. Этот документ описывает, как быстро начать.

## Кодекс поведения

Участвуя в проекте, вы соглашаетесь соблюдать [Кодекс поведения](CODE_OF_CONDUCT.md).

## С чего начать

1. Сделайте **fork** и клонируйте репозиторий.
2. Установите зависимости и поднимите локальную БД:
   ```bash
   npm install
   cp .env.example .env
   npm run docker:db:up
   npm run db:migrate
   npm run db:seed        # опционально
   npm run dev            # http://localhost:3000
   ```
3. Создайте ветку: `git checkout -b feat/краткое-описание` (или `fix/...`, `docs/...`).

Подробнее об архитектуре и структуре — в [README](README.md). Для десктоп-сборки см.
[docs/WINDOWS_DESKTOP.md](docs/WINDOWS_DESKTOP.md), для веб-деплоя —
[docs/WEB_DEPLOY.md](docs/WEB_DEPLOY.md).

## Перед коммитом — гейты

Все проверки должны быть зелёными (их же гоняет CI и pre-commit хук):

```bash
npm run typecheck      # tsc --noEmit, без ошибок типов
npm run lint           # ESLint, 0 предупреждений
npm run test           # Vitest
npm run build:web      # веб-сборка
npm run build:static   # статический экспорт (десктоп) — не должен ломаться
```

> Pre-commit хук (Husky + lint-staged) автоматически прогонит ESLint/Prettier и
> связанные тесты. Не отключайте хуки (`--no-verify`) без необходимости.

## Правила кода

- **TypeScript строгий** — без `as any` и подавления ошибок. Валидация ввода через **Zod**.
- **Двойной режим.** Доступ к данным идёт только через интерфейс `ApiClient`
  (`lib/api/`). Серверная логика — в роутах `app/api/**/route.web.ts` (web-only,
  не попадают в десктоп-экспорт). Не обращайтесь к Prisma из клиентских компонентов.
- **Изоляция по пользователю (web).** Любой серверный запрос к данным фильтруется по
  текущему `userId` (см. `lib/auth/current-user.ts`). Новые сущности — с тем же правилом.
- **Чистые функции — с тестами.** Новая бизнес-логика (расчёты, парсеры, пресеты)
  обязательно покрывается unit-тестами в `tests/`.
- **Стиль** — следуйте окружающему коду: те же имена, идиомы, плотность комментариев.
  Форматирование — Prettier (применяется автоматически).

## Сообщения коммитов

Используем [Conventional Commits](https://www.conventionalcommits.org/):
`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`, `perf:`. Пример:

```
feat(import): пресет CSV для Альфа-Банка
fix(web): риск-профиль не отображался в настройках
```

## Pull Request

1. Убедитесь, что все гейты зелёные.
2. Опишите **что** и **зачем**; приложите скриншоты для изменений UI.
3. Свяжите связанные issue (`Closes #123`).
4. Заполните [шаблон PR](.github/PULL_REQUEST_TEMPLATE.md).

Поддерживаем небольшие сфокусированные PR — их проще ревьюить и мёржить.

## Сообщить о баге или предложить фичу

Используйте шаблоны: [баг-репорт](.github/ISSUE_TEMPLATE/bug_report.yml) и
[предложение фичи](.github/ISSUE_TEMPLATE/feature_request.yml). О проблемах
безопасности — **не** через публичные issue, см. [SECURITY.md](SECURITY.md).
